# Agent Guide

## Repository Role

`prism-sync` is the standalone sync engine for
[Prism](https://github.com/prismplural/prism-app). It contains Rust crates for
encrypted CRDT sync, Dart/Flutter FFI packages generated by
`flutter_rust_bridge`, and a self-hostable relay server.

This repository is public and standalone. Do not assume it is checked out inside
a monorepo. Public instructions should work from a plain clone. Maintainer-only
multi-repo workflow, release coordination, and private operations notes belong
outside this repository.

The relay stores opaque encrypted blobs. Plaintext app data and long-term key
material must stay client-side.

## Related Public Repositories

- `prism-app`: Flutter app that consumes the Dart packages in this repo.
- `prism-fronters`: public PluralKit fronters dashboard.

If a change spans sync and app, prefer a backward-compatible sync change first,
then update `prism-app` to consume the new sync revision. Keep PRs and commits
repo-local.

## Build And Test

Rust MSRV is `1.75`. The workspace uses Rust 2021.

```bash
cargo build --workspace
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
cargo fmt --all
```

The `justfile` wraps common commands:

```bash
just build
just test
just test-crate prism-sync-core
just lint
just fmt
just fmt-check
```

Per-crate checks:

```bash
cargo test -p prism-sync-crypto
cargo test -p prism-sync-core
cargo test -p prism-sync-relay
cargo test -p prism-sync-crypto --test cross_language_vectors -- --ignored
```

Run cross-language vector tests when touching crypto, serialization formats, or
the FFI boundary.

## Dart Packages And FFI

The Dart workspace lives under `dart/packages/` and is managed with Melos:

```bash
cd dart
dart pub global activate melos
melos bootstrap
melos run test
```

After changing the public FFI API in `crates/prism-sync-ffi/src/api.rs`, run:

```bash
flutter_rust_bridge_codegen generate
```

This regenerates `dart/packages/prism_sync/lib/generated/`. Never edit generated
bindings by hand. Commit Rust API changes and regenerated Dart package files
together.

Consumers can depend on the Dart packages through git, or use local
`pubspec_overrides.yaml` path overrides while developing against a sibling clone.

## Project Shape

```text
crates/
├── prism-sync-crypto/   # Standalone crypto primitives
├── prism-sync-core/     # CRDT engine, storage, pairing, relay client
├── prism-sync-ffi/      # flutter_rust_bridge API for Dart/Flutter
├── prism-sync-relay/    # Self-hostable relay server
└── prism-sync-bench/    # Criterion benchmarks

dart/packages/
├── prism_sync/          # Generated Dart bindings and Cargokit integration
├── prism_sync_drift/    # Drift adapter
└── prism_sync_flutter/  # Flutter secure storage and provider glue

self-host/               # Docker, Compose, Kubernetes, and runbook docs
docs/                    # Protocol and design docs
```

Crate dependency flow:

```text
prism-sync-crypto -> prism-sync-core -> prism-sync-ffi

prism-sync-relay is standalone.
```

## Core Engineering Rules

- Keep `prism-sync-crypto` independent from sync state and application concepts.
- Keep `SyncStorage` and `SyncRelay` object-safe. Do not add generic methods to
  those traits.
- Wrap blocking storage work in the engine with `tokio::task::spawn_blocking`.
- FFI functions should use primitive, bridge-friendly types. Do not expose trait
  objects across the FFI boundary.
- Merge order is field-level last-write-wins with the protocol tiebreaker
  `HLC -> device_id -> op_id`.
- Relay paths are versioned under `/v1/sync/{sync_id}/...`.
- Pull should skip unverifiable batches from unknown, deleted, or invalid
  senders rather than aborting the whole sync.
- Device identity uses per-device CSPRNG secrets, never shared DEK-derived
  identity keys.

## Crypto And Secret Handling

- Store sensitive key material in `Zeroizing<Vec<u8>>` or equivalent zeroizing
  wrappers.
- Do not log plaintext records, DEKs, MEKs, device secrets, session tokens,
  invite secrets, raw wrapped key material, or decrypted relay payloads.
- Batch signatures use hybrid Ed25519 plus ML-DSA-65 over the deterministic
  protocol V3 canonical format.
- X-Wing / ML-KEM / ML-DSA changes need vector tests and compatibility review.
- Side-channel residual risks and threat-model details belong in public protocol
  or architecture docs when they affect users or auditors.

## Relay And Self-Hosting

The relay is self-hostable and should remain app-agnostic. See:

- `self-host/SELF-HOSTING.md`
- `crates/prism-sync-relay/README.md`
- `ARCHITECTURE.md`

Do not put private deployment details, internal hostnames, unpublished operations
notes, or maintainer-only release procedures in this repository.

## Public Documentation Boundary

Keep public docs focused on building, testing, contributing, protocol behavior,
self-hosting, and auditing. Private multi-repo workspace instructions,
maintainer coordination, and unreleased product strategy should live outside
this public repository.
