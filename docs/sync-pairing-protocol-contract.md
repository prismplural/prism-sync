# Sync Pairing Protocol Contract

**Status:** Refreshed 2026-04-08 — compact QR/URL transport removed; pairing now uses relay-based ceremony with SAS verification

This document describes the current sync-pairing and registration wire shape,
including the compatibility fields that still exist during rollout.

This document originated as the client-owned contract freeze during the earlier
Agent A / Agent B split. It is now kept as a general protocol reference rather
than a live ownership boundary.

## Source Files

- `sync/crates/prism-sync-core/src/pairing/models.rs` — PairingRequest, PairingResponse, SignedRegistrySnapshot
- `sync/crates/prism-sync-core/src/relay/traits.rs` — RegisterRequest, RegistryApproval, SignedInvitationPayload

## 1. PairingRequest (Joiner → Approver) — REMOVED

> **Removed:** The compact QR-based `PairingRequest` transport has been replaced by the relay-based pairing ceremony with SAS verification. The JSON shape is retained here for historical reference only.

### JSON Shape (historical)

```json
{
  "device_id": "string (12 hex chars)",
  "ed25519_public_key": [/* 32 bytes */],
  "x25519_public_key": [/* 32 bytes */]
}
```

### Compact Binary (QR transport, version 0x01) — REMOVED

```
[1B  version = 0x01]
[2B  device_id len][device_id UTF-8]
[32B ed25519_public_key]
[32B x25519_public_key]
```

## 2. PairingResponse (Approver → Joiner) — compact transport REMOVED

Sent by the existing trusted device after approving a PairingRequest.
The JSON shape is still used internally by the relay-based ceremony, but the compact binary encoding and QR/URL transport have been removed.

### JSON Shape

```json
{
  "relay_url": "wss://...",
  "sync_id": "string (64 hex chars = 32 bytes)",
  "mnemonic": "12 BIP39 words",
  "wrapped_dek": [/* 72 bytes: 24B nonce + 32B ciphertext + 16B MAC */],
  "salt": [/* 32 bytes: Argon2id salt */],
  "signed_invitation": "hex(HybridSignature: Ed25519 + ML-DSA-65)",
  "signed_keyring": [/* SignedRegistrySnapshot wire format */],
  "inviter_device_id": "string",
  "inviter_ed25519_pk": [/* 32 bytes */],
  "joiner_device_id": "string | null",
  "current_epoch": 0,
  "epoch_key": [/* 0 or 32 bytes */],
  "registry_approval_signature": "hex(HybridSignature: Ed25519 + ML-DSA-65) | null"
}
```

`registry_approval_signature` is present for existing-group approvals and absent
for first-device bootstrap and older compatibility flows.

### Compact Binary (QR/URL transport) — REMOVED

The compact binary encoding (`to_compact_bytes` / `from_compact_bytes`) and the QR/URL transport have been removed. Pairing now uses the relay-based ceremony with SAS verification, which transmits pairing data as JSON through relay slots.

### Admission Context

`PairingResponse::admission_context()` returns:
- `"first_device"` — snapshot has exactly 1 entry matching `inviter_device_id`
- `"existing_group"` — snapshot has multiple entries or entry doesn't match inviter

## 3. SignedRegistrySnapshot

Wire format embedded in `PairingResponse.signed_keyring`.

### Wire Format

```
[HybridSignature bytes (Ed25519 + ML-DSA-65)]
[variable: canonical JSON]
```

Signature covers: `PRISM_SYNC_REGISTRY_V1\x00` || canonical JSON.

### Canonical JSON

Entries sorted by `device_id`, then serialized as a JSON array:

```json
[
  {
    "sync_id": "string",
    "device_id": "string",
    "ed25519_public_key": [/* 32 bytes */],
    "x25519_public_key": [/* 32 bytes */],
    "status": "active" | "revoked"
  }
]
```

### Verification

The joiner must:
1. Split at HybridSignature boundary → signature + JSON
2. Rebuild signing data: `PRISM_SYNC_REGISTRY_V1\x00` || JSON
3. Verify hybrid signature (Ed25519 + ML-DSA-65) against inviter keys
4. Deserialize entries and import via `DeviceRegistryManager::import_keyring`

## 4. RegisterRequest (Client → Relay)

### Two Admission Paths

| Field | First Device (new group) | Existing Group (joiner) |
|-------|--------------------------|------------------------|
| `device_id` | Generated by creator | From PairingRequest or pre-generated |
| `signing_public_key` | 32B Ed25519 | 32B Ed25519 |
| `x25519_public_key` | 32B X25519 | 32B X25519 |
| `ml_dsa_65_public_key` | ML-DSA-65 PQ signing key | ML-DSA-65 PQ signing key |
| `ml_kem_768_public_key` | ML-KEM-768 PQ KEM key | ML-KEM-768 PQ KEM key |
| `registration_challenge` | HybridSignature (Ed25519 + ML-DSA-65) over challenge data | HybridSignature (Ed25519 + ML-DSA-65) over challenge data |
| `nonce` | From `get_registration_nonce()` | From `get_registration_nonce()` |
| `signed_invitation` | Present (self-signed) | Optional compatibility fallback |
| `pow_solution` | Required if relay demands it | Optional |
| `first_device_admission_proof` | Optional platform signal | `None` |
| `registry_approval` | `None` | Present (approver attestation) |

### Registration nonce response

```json
{
  "nonce": "string",
  "pow_challenge": {
    "algorithm": "sha256_leading_zero_bits",
    "difficulty_bits": 18
  },
  "admission_hints": {
    "play_integrity_cloud_project_number": 123456789
  }
}
```

`pow_challenge` and `admission_hints` are both optional.

### FirstDeviceAdmissionProof JSON

```json
{
  "kind": "android_key_attestation",
  "certificate_chain": ["base64(DER cert)", "base64(DER cert)"],
  "play_integrity_token": "string | null"
}
```

```json
{
  "kind": "apple_app_attest",
  "key_id": "string",
  "attestation_object": "base64(CBOR)"
}
```

### Challenge Data Format

```
"PRISM_SYNC_CHALLENGE_V1" || 0x00
|| len_prefixed(sync_id)
|| len_prefixed(device_id)
|| len_prefixed(nonce)
```

Where `len_prefixed` = `[4B big-endian length][data]`.

### RegistryApproval Shape

```json
{
  "approver_device_id": "string",
  "approver_ed25519_pk": "hex(32 bytes)",
  "approval_signature": "hex(64 bytes)",
  "signed_registry_snapshot": [/* wire format bytes */]
}
```

### SignedInvitationPayload Shape

```json
{
  "sync_id": "string",
  "relay_url": "string",
  "wrapped_dek": "hex",
  "salt": "hex",
  "inviter_device_id": "string",
  "inviter_ed25519_pk": "hex(32 bytes)",
  "signature": "hex(64 bytes)",
  "joiner_device_id": "string | null",
  "current_epoch": 0,
  "epoch_key_hex": "hex (empty if epoch 0)"
}
```

### Compatibility note

The long-term direction is to remove sync-pairing dependence on
`signed_invitation` for existing-group admission. The shipped contract still
allows it as a rollout fallback, so current docs and tests must treat
`registry_approval` as the preferred path and `signed_invitation` as temporary
compatibility.

## 5. Error Codes Flutter Should Expect

From `CoreError` in `sync/crates/prism-sync-core/src/error.rs`:

| Error | When | Flutter action |
|-------|------|---------------|
| `Engine("invalid pairing response: ...")` | Epoch field validation fails | Show "invalid invite" |
| `Engine("invitation signature invalid: ...")` | Tampered PairingResponse | Show "invite tampered" |
| `Engine("registry snapshot rejected: ...")` | Tampered signed_keyring | Show "invite tampered" |
| `Crypto(...)` | Wrong password / corrupt DEK | Show "wrong password" |
| `CoreError::DeviceKeyChanged { device_id }` | Key pinning violation | Show "device key changed" warning |
| `CoreError::Relay { kind: DeviceIdentityMismatch, ... }` | Re-register or sender key mismatch | Show re-pair / export-safe recovery |
| `CoreError::Relay { kind: DeviceRevoked, remote_wipe, ... }` | This device was revoked | Show terminal revoked-device handling |
| `CoreError::Relay { kind: Server, status_code: Some(429), ... }` | Too many attempts | Show "rate limited" |
| `Engine("unsupported first-device admission challenge: ...")` | Unknown PoW algorithm | Show "update required" |

## Backward Compatibility

- Compact binary encoding for `PairingRequest` and `PairingResponse` has been removed; pairing uses the relay-based ceremony exclusively
- new fields use `#[serde(default)]`, so older responses decode with defaults
- `signed_keyring` wire format: `[HybridSignature bytes (Ed25519 + ML-DSA-65) || JSON array]`
- `signed_invitation` remains in the register contract as a temporary
  compatibility fallback for existing-group admission during rollout
- `pow_challenge`, `admission_hints`, and `play_integrity_token` are all
  optional
